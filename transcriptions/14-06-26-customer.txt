SUMMARY — Key takeaways
------------------------------------------------------------

1. ARCHITECTURE LAYERS CONFIRMED AS CORRECT
   The customer independently arrives at the same three-layer split we already
   designed:
     - "Testing"   = runs purely on historical data, no broker connection
                      (our Simulation Engine / b_Test)
     - "Simulation" = connects to the broker/sandbox and actually places
                      orders, but on near-real-time / fresh data, no real
                      money (our Trading Bot)
     - "Real trading" = same code, only the API token changes from sandbox
                      to production. Explicitly OUT OF SCOPE for this
                      project, but the architecture should not block it.
   No architectural change needed. Recommendation: document this mapping
   explicitly (testing -> Simulation Engine, simulation -> Trading Bot,
   real trading -> out of scope but token-swappable).

2. ARCHITECTURE = ANSWER TO NON-FUNCTIONAL REQUIREMENTS, NOT A PICTURE
   The customer spent significant time explaining that architecture is not a
   diagram — it's the set of decisions that guarantee non-functional
   requirements (availability, flexibility, reliability, etc.) are met.
   Concrete example given: the Broker/Data Adapter pattern exists
   specifically to satisfy a FLEXIBILITY requirement (ability to switch
   brokers/data sources cheaply when conditions change, e.g. switching
   from T-Bank to another provider due to fee increases).
   Action: in our architecture doc, explicitly map each major pattern
   (Broker Adapter, Strategy plugin, etc.) to the NFR it satisfies.

3. THREE ARCHITECTURAL VIEWS RECOMMENDED
   Standard split into:
     - Static view      -> module dependencies, what functionality lives
                            where (e.g. class diagram)
     - Dynamic view      -> runtime interaction between modules, message
                            order/sequence
     - Deployment view   -> where things run, how they communicate over
                            the network
   Currently everything is mixed into one diagram. Not urgent, but
   splitting into these three views would make future presentations to
   the customer/customer look more professional.

4. COUPLING / COHESION — GENERAL PRINCIPLE, ALREADY FOLLOWED
   customer reiterates the standard "high cohesion, low coupling" idea — group
   related functionality together, minimize dependencies between modules,
   make changes localized. This validates the modular-monolith +
   interfaces.py approach we already have.

5. STRATEGY VISUALIZATION / "STRATEGY GENERATOR" IDEA (OPTIONAL, FUTURE)
   The customer floated an idea for a possible future feature: a visual
   "strategy generator" UI where users connect signals/indicators with
   probabilities/weights/thresholds to build strategies graphically,
   built on top of the existing list of implemented strategies/signals.
   Explicitly framed as optional / a "maybe", not core scope. Worth
   noting as a stretch goal for later MVPs.

6. PROCESS FEEDBACK — BACKLOG / BOARD HYGIENE
   The customer reviewed the GitHub board and noted some issues were "in
   progress" but not updated/reviewed. Concrete ask: keep board status
   current (move items to review, get sign-off from Georgii/team lead),
   add a milestone filter for "this week", and send a weekly report
   (deadline mentioned: Monday 8pm).

7. SEQUENCING ADVICE: PROTOTYPE FIRST, POLISH ARCHITECTURE AFTER
   Strong, repeated advice: don't over-invest in a "perfect" architecture
   before a working MVP exists. Build something minimal that works (even
   just "buy and sell"), show it to the customer, learn what the customer
   actually meant, THEN refine the architecture for flexibility/
   robustness. Rationale: if the team perfects architecture first and the
   customer's mental model differs once they see results, both the
   architecture AND the already-built code need rework — more total time
   lost than iterating from a rough prototype.
   Practical implication for us: it's fine (expected, even) if the
   customer's reaction at the week-4 demo leads to architecture tweaks —
   this is normal, not a failure signal.
   Suggested workflow: split work — part of the team builds a rough
   working solution, part works on the "correct" structure/architecture,
   then merge the two, synchronizing regularly.

8. ADVICE: PROPOSE SOLUTIONS WITH ALTERNATIVES + RATIONALE
   customer explicitly praised the approach of not just asking the customer
   open questions, but proposing a solution with alternatives and
   reasoning ("we considered X, Y, Z; we recommend X because..."). This
   matches the "open questions for customer" document style we've been
   using — continue with it (e.g. for the backtester structure draft the
   customer suggested sending over for comments).

9. CAREER / MOTIVATION CONTEXT (LOW PRIORITY, FYI)
   customer mentioned that hands-on experience with trading-system architecture
   is valuable if the team later wants to work in finance/banking (e.g.
   T-Bank), since that sector values people with domain experience plus
   demonstrated architectural reasoning (not just "vibe-coded" solutions).
   Not actionable, just context/motivation.

    customer expects to see a working MVP ("something that runs") at the next
    meeting, not just plans/diagrams.
