BackTestBench — First Customer Meeting Transcript

------------------------------------------------------------
SUMMARY — Key takeaways
------------------------------------------------------------

1. TEAM & ROLES INTRODUCED
   8-person team with roles: Business Analyst (Liubov), System
   Architect/Tech Lead (Georgii), Data Engineer (Bogdan), Strategy Module
   Developer (Philip), Core Engine Developer (Samy), Analytics/Sentiment
   Developer (Maxim), Frontend Developer (Marat), DevOps/QA (Nikolay).

2. PROJECT GOAL
   Build a flexible, modular backtesting and simulation tool for trading
   strategies — explicitly NOT a production trading bot. Focus is on
   engineering qualities (flexibility, reliability, modularity), not on
   making money. Customer repeatedly disclaims any promotion of real
   trading due to risk.

3. TWO SEQUENTIAL STAGES
   - Backtesting: run strategies against historical data, compute
     metrics (P&L, etc.), rank strategies.
   - Sandbox/live validation: run the selected top strategies against
     fresh data via T-Bank sandbox (or similar), without real money.
   These are sequential, not mutually exclusive modes.

4. KEY ARCHITECTURAL ELEMENTS DISCUSSED
   - Broker Adapter: abstraction layer so the system can work with
     different brokers/providers (T-Bank, Binance, MOEX, etc.) without
     rewriting core logic. Motivated by flexibility (commissions/
     conditions can change between providers).
   - Strategy definition: leaning toward config files (JSON/YAML) or
     simple functions rather than a custom DSL or raw Python scripts the
     system dynamically loads — customer is open but wants flexibility
     without rewriting the core for every new strategy.
   - Indicators / filters / actions as composable building blocks (MA,
     EMA, RSI thresholds, time filters, risk filters, stop-loss/take-
     profit, trailing stops).
   - Risk management: position sizing, hedging (e.g. oil vs silver
     correlation), risk/profit ratios, stop-loss/take-profit, trailing
     stops.
   - Data sources: historical data via T-Bank Investments sandbox API
     (or others); candles vs. tick/trade-level data; multiple instruments
     (oil, gold, equities, crypto derivatives via T-Bank).
   - Performance: backtesting many strategies x instruments x parameter
     combinations is computationally heavy (potentially many hours on a
     laptop). Vectorization, GPU (e.g. A100) discussed as future options,
     but explicitly NOT required for MVP — Python is fine, no real-time
     latency requirements.

5. SCHEDULED RECALCULATION / TOP-N CONCEPT
   The customer described a loop: run backtests across many
   strategies/instruments/parameters -> rank by P&L (compared against a
   baseline deposit rate, ~13% annual) -> take top-N -> run those as bots
   on fresh data -> periodically (e.g. every 1-2 weeks) recalculate and
   update the top-N list. This recalculation should run automatically
   without manual intervention.

6. NON-FUNCTIONAL REQUIREMENTS EMPHASIZED
   - Fault tolerance: a failed backtest/recalculation run should not
     break the system or silently corrupt the top-N list; notifications
     (e.g. email) should be sent on failure.
   - Reliability/testing: customer stresses this is closer to a
     "mission-critical" system (bugs could cause large simulated losses
     if ever connected to real trading) — testing and QA are important,
     and the team is missing a dedicated QA role (later addressed —
     DevOps/QA role added).
   - The system should support adding new strategies/signal sources
     (news sentiment, ML pattern prediction, whale tracking) without
     rewriting the core — flagged as optional/"starred" advanced
     features.

7. OPTIONAL / ADVANCED FEATURES (marked with a star in the project
   description, lower priority)
   - News sentiment analysis (weighted signal contributions).
   - ML-based pattern prediction (e.g. head-and-shoulders probability).
   - "Whale" / large-order tracking as a signal.
   - Multi-broker support beyond T-Bank (Binance, MOEX, US equities via
     derivatives).

8. FRONTEND / VISUALIZATION
   Customer explicitly said a complex custom UI is NOT mandatory if it
   doesn't add immediate value — Grafana or similar tools on top of the
   database are acceptable alternatives, especially for MVP. Lean
   development emphasized: don't build what you don't need yet.

9. SCOPE FOR MVP (2 WEEKS)
   Customer strongly pushed for a drastically reduced initial scope:
   - One simple strategy
   - One instrument
   - One bot/instance
   - Just "buy and sell" as a starting point, then iterate.
   Goal: working MVP within ~2 weeks to validate direction before
   investing more in architecture/breadth.

10. PROCESS / TEAM MANAGEMENT
    - Customer raised concerns about coordinating an 8-person team
      (communication overhead grows roughly with team size squared);
      suggested smaller sub-groups and clear ownership.
    - Recommended agile/scrum-style process with weekly demos, each
      sprint increasing product value.
    - Customer is available roughly weekly, flexible on day (next meeting
      also proposed for Sunday). Asked the team to set up a Telegram
      group for async communication and possibly a smaller in-person
      follow-up (team lead + a couple of analysts).
    - Customer is open to giving feedback on a written requirements doc
      / Google Doc asynchronously between meetings.
    - Customer is also course staff in some capacity (was a developer for
      ~7 years) and is available for technical/organizational questions
      between meetings — doesn't need to wait for the next scheduled
      meeting for important blockers.

------------------------------------------------------------
OPEN QUESTIONS RAISED DURING THE MEETING (worth tracking)
------------------------------------------------------------
- Strategy definition format: config-driven (JSON/YAML/text files) vs.
  dynamically-loaded Python scripts vs. form-based parameters — customer
  leans config-driven but open to discussion; final call left to the
  team based on what's feasible without constant core rewrites.
- Sentiment analysis scope: mock news feed to prove architecture, vs.
  real integration with a live financial news API — customer suggested
  starting by checking which historical news affected which instruments,
  i.e. start retrospective/historical rather than live.
- Recalculation frequency: customer's instinct is ~1-2 weeks by default,
  but explicitly says "we'll see with prototypes" and wants it
  configurable.
- Whether to compare strategy P&L against a baseline deposit rate (~13%
  annual) as a go/no-go filter for running a strategy live — customer
  leans yes but flags it's not a fully settled criterion yet.
- Handling of repeated identical signals (e.g. multiple consecutive "buy"
  signals) — does the bot buy multiple times, ignore subsequent signals,
  or adjust position size? Customer flags this as a real design question
  for the Strategy/Core Engine modules, not yet resolved.
- Sandbox vs. real account API differences in T-Bank Investments — token
  swap may now make sandbox and real API nearly identical (customer
  mentions a recent API version change but hasn't verified
  compatibility).

------------------------------------------------------------
BOTTOM LINE
------------------------------------------------------------
This meeting establishes the foundational scope, team roles, and
customer's mental model: a modular backtesting/sandbox platform with a
Broker Adapter for flexibility, config-driven strategies, scheduled
recalculation producing a ranked top-N list, and a strong push toward a
minimal 2-week MVP (one strategy, one instrument, one bot) before
expanding scope. Most "advanced" features (sentiment, ML, whale tracking,
multi-broker, custom frontend) are explicitly optional / post-MVP.
